2  Meteor2 documentation

2.1 Background

A comprehensive understanding of microbial communities requires the integration of taxonomic, functional, and strain-level profiling (TFSP). In shotgun metagenomics, these three layers are tightly connected: taxonomy identifies who is there, functional profiling estimates what they can do, and strain-level analysis provides insight into within-species variation and population structure.

Most reference-based metagenomic profilers rely on one of three broad strategies:

  1. whole-genome reference databases,
  2. species-specific marker genes, or
  3. universal single-copy marker genes.

These approaches are often highly effective for taxonomic assignment, but they usually provide only indirect or partial access to functional interpretation. In practice, this means that taxonomic and functional profiling are frequently performed with separate tools and databases, which can increase workflow complexity and create inconsistencies between the two outputs.

This limitation is particularly evident in approaches designed for broad applicability across many ecosystems. Because these methods often depend on conserved marker genes, they gain generalizability at the cost of reduced sensitivity to ecosystem-specific functional variation. As a result, functional characterization often requires additional annotation workflows based on pan-genome, orthology, or pathway resources.

An alternative strategy is to use ecosystem-specific microbial gene catalogues. These catalogue-based approaches rely on smaller but highly informative reference collections and enable a more coherent integration of taxonomic and functional profiling. This is the conceptual framework on which Meteor2 is based (ghozlane2025meteor2?).

2.2 What is Meteor2?

Meteor2 is a tool for quantitative metagenomic profiling of complex ecosystems based on microbial gene catalogues. Rather than relying primarily on universal marker genes, Meteor2 quantifies genes by mapping reads against ecosystem-specific reference catalogues and derives taxonomic, functional, and strain-level profiles from the same reference framework (ghozlane2025meteor2?).

Its main analytical unit is the Metagenomic Species Pan-genome (MSP), originally introduced with MSPminer (plazaonate2019mspminer?). In this framework, genes are grouped into microbial species-level units based on co-abundance patterns across samples. Within each MSP, a subset of highly connected genes is defined as signature genes, which are used as the most robust indicators for MSP detection and quantification.

Meteor2 uses microbial gene catalogues to perform:

  • species-level taxonomic profiling (Bacteria, Archaea, and Eukaryotes),
  • functional profiling,
  • strain-level population structure inference.

In addition, Meteor2 integrates several functional annotation repertoires into its catalogues, including:

  • KEGG Orthologs (KO) from KEGG database (kanehisa2016kegg?),
  • carbohydrante-active enzymes (CAZymes) from dbCAN (yuan2023dbcan?),
  • antibiotic resistance genes (ARGs) from MUSTARD (roux2019mustard?).

These annotations can be quantified either at the individual gene level or at the MSP level, thereby linking taxonomic composition to functional potential within a unified framework.

2.3 Microbial gene catalogues

Meteor2 currently supports 10 curated microbial gene catalogues, each tailored to a specific host-associated ecosystem (ghozlane2025meteor2?). These catalogues cover:

  • human microbiomes: gut, oral, and skin,
  • domestic animals: cat and dog,
  • laboratory models: mouse and rat,
  • livestock: chicken, pig, and rabbit.

Each catalogue includes two main components:

  1. MSPs, in which genes are clustered by co-abundance;
  2. unclustered genes, which may represent biological content that is not captured by metagenome-assembled genomes (MAGs) or stable MSP structures (coelho2022biogeography?).

Meteor2 also incorporates three types of higher-level functional module systems:

  • Gut Brain Modules (GBMs) (vallescolomer2019neuroactive?),
  • Gut Metabolic Modules (GMMs) (vieirasilva2016species?),
  • KEGG modules (kanehisa2012keggmapper?).

For rapid analysis, Meteor2 additionally provides a light / fast mode, based on a reduced catalogue containing only 100 signature genes per MSP. These genes correspond to the most connected genes within each MSP and allow accelerated taxonomic and strain-level profiling. Functional profiling is not available in this reduced mode (ghozlane2025meteor2?).

2.4 Overview of the workflow

Meteor2 estimates the abundance of genes, MSPs, and functions within microbial communities, and also reconstructs sample-specific strain variation.

The analytical workflow consists of four main steps (ghozlane2025meteor2?):

  1. Download an ecosystem-specific microbial gene catalogue
  2. Align short metagenomic reads to the catalogue
  3. Compute abundance profiles for genes, MSPs, and functions
  4. Reconstruct sample-specific MSP genotypes and estimate mutation rates

The pipeline starts by aligning metagenomic reads against a reference microbial gene catalogue using Bowtie2 (langmead2012bowtie2?). From the mapping results, Meteor2 derives gene abundance estimates and propagates these measurements to species- and function-level summaries. In parallel, a strain branch performs SNV calling and reconstructs sample-specific consensus sequences for downstream phylogenetic and distance-based analyses.

2.5 Quantification modes

After read alignment, Meteor2 re-estimates gene counts using one of three counting modes:

  • unique: counts only reads with a single valid alignment.
  • total: assigns all alignments to all matched genes and sums them directly.
  • shared: redistributes multi-mapping reads proportionally across the genes they align to, based on the genes’ unique read counts. If all unique counts are zero, the read is distributed evenly across all compatible genes.

The shared mode is particularly useful when reads map to homologous genes and a proportional assignment strategy is preferred over strict unique counting.

2.6 Taxonomic profiling

Meteor2 derives taxonomic profiles from gene count tables through two main operations:

  1. normalization,
  2. reduction from genes to MSPs.

2.6.1 Normalization

Meteor2 supports the following normalization strategies:

  • Depth coverage: read count divided by gene length and multiplied by 100, with gene length corrected for edge effects due to read length (metahit2014catalog?).
  • FPKM (Fragments Per Kilobase Million).

2.6.2 Reduction to MSP abundance

MSP abundance is estimated by averaging the abundance of signature genes after normalization. By default, Meteor2 considers the 100 most central signature genes per MSP.

An MSP is assigned an abundance of 0 if too few signature genes are detected in the sample. In standard catalogues, the default requirement is more stringent; in light catalogues, the detection threshold is reduced, with a default requirement of 20% of signature genes.

This approach provides a robust species-level abundance estimate while minimizing noise from poorly supported genes.

2.7 Functional profiling

Meteor2 computes the abundance of functional entities in two complementary ways:

  • by aggregating the abundance of genes annotated with a given function,
  • or by aggregating the abundance of MSPs carrying that function.

When MSP-based functional inference is used, only genes that are actually detected in the sample (at least one mapped read) are considered.

2.7.1 Functional modules

Meteor2 supports module-level profiling for systems such as GBMs, GMMs, and KEGG modules. For a given module, an MSP is considered a module carrier when the module completeness exceeds 90% in that sample (ghozlane2025meteor2?).

Because some module steps can be satisfied by alternative functions, Meteor2 evaluates all compatible alternative configurations and retains the maximum completeness value. The total abundance of all MSP carriers defines the module potential in that sample, together with the list of contributing taxa.

This strategy provides an explicit bridge between community composition and functional potential, making it possible to identify not only which functions are present, but also which species are likely contributing to them.

2.8 Strain-level analysis

Meteor2 performs strain-level analysis in three main stages:

  1. SNV calling using FreeBayes (garrison2012freebayes?)
  2. Reconstruction of sample-specific MSP sequences
  3. Pairwise comparison and phylogenetic analysis of reconstructed MSPs

2.8.1 1. SNV calling

Meteor2 identifies single nucleotide variants from mapped reads and uses them to build a sample-specific consensus gene catalogue.

2.8.2 2. Reconstruction of sample-specific MSPs

Only MSPs with sufficient coverage are retained for downstream strain analysis. By default, at least 80 out of 100 signature genes must have a minimum breadth coverage of 0.5.

2.8.3 3. Mutation rate and distance estimation

MSPs detected in at least two samples are considered for mutation-rate analysis. By default, at least 10 informative sites are required to estimate distances. Pairwise distances between sample-specific MSP sequences are then computed under a General Time Reversible (GTR) evolutionary model, allowing strain similarity and divergence to be quantified (ghozlane2025meteor2?).

Meteor2 can also generate phylogenetic trees from these reconstructed sequences.

2.9 Installation

Meteor2 can be installed with conda, including its dependencies:

conda create --name meteor -c conda-forge -c bioconda meteor

Alternatively, it can be installed with pip using a recent Python version:

pip install meteor

To test the installation:

meteor test

2.10 Nextflow wrapper

Meteor2 also provides a Nextflow wrapper (nf-meteor.nf) for automated execution of the full workflow (meteorgithub?).

nf-meteor.nf --in <fastq_dir> --catalogue_name <catalogue_name> --out <output_dir> --cpus <nb_cpus> -w <temp_work_dir>

2.10.1 Main parameters

  • --in: directory containing paired fastq.gz files
  • --out: output directory
  • --cpus: number of CPUs to use
  • --catalogue_name: name of a prebuilt catalogue
  • --catalogue: path to a custom catalogue (overrides --catalogue_name)
  • --fast: enable fast mode (no functional analysis)
  • --check_catalogue: check whether the catalogue is compatible with the input reads

Allowed --catalogue_name values include:

  • fc_1_3_gut
  • gg_13_6_caecal
  • clf_1_0_gut
  • hs_10_4_gut
  • hs_8_4_oral
  • hs_2_9_skin
  • mm_5_0_gut
  • oc_5_7_gut
  • rn_5_9_gut
  • ssc_9_3_gut

2.11 Getting started

A basic Meteor2 workflow typically involves the following steps:

  1. Download or build a reference catalogue
  2. Import and index FASTQ files
  3. Map reads against the reference catalogue
  4. Compute taxonomic and functional abundance profiles
  5. Perform strain-level profiling
  6. Merge outputs across samples when needed

2.12 1. Download a reference catalogue

Meteor2 requires a locally available microbial gene catalogue, either in full or light format.

  • The full catalogue contains all genes and supports taxonomic, functional, and strain-level profiling.
  • The light catalogue contains only marker/signature genes and supports rapid taxonomic and strain-level profiling, but not functional profiling.

Download a full catalogue with:

meteor download -i <name> -c -o <refdir>

Download a light catalogue with:

meteor download -i <name> -c --fast -o <refdir>

2.13 2. Import FASTQ files

Meteor2 requires an initial FASTQ indexing step:

meteor fastq -i <fastqdir> [-p paired reads] -o <outputdir>

When multiple sequencing files are available for the same library, the -m option can be used to group them automatically.

For example, if the directory contains:

Illumina_lib1-SAMPLE_01.fastq
Illumina_lib1-SAMPLE_02.fastq
Illumina_lib2-SAMPLE_01.fastq
Illumina_lib2-SAMPLE_02.fastq

The following command will group files belonging to the same sample/library structure:

meteor fastq -i ./ -m SAMPLE_\\d+ -o outputdir

2.14 3. Mapping

FASTQ files are mapped against a reference catalogue to generate gene count tables:

meteor mapping -i <fastqdir/sampledir> -r <refdir> -o <mappingdir>

Before this step, it is recommended to remove:

  • low-quality reads,
  • short reads (e.g. < 60 nt),
  • host-derived reads.

2.15 4. Taxonomic and functional profiling

Genes in the catalogue are clustered into Metagenomic Species Pan-genomes (MSPs) with MSPminer (plazaonate2019mspminer?) and functionally annotated against:

  • KEGG (kanehisa2016kegg?),
  • dbCAN (yuan2023dbcan?),
  • MUSTARD (roux2019mustard?).

MSP and functional profiles are computed from the gene count table with:

meteor profile -i <mappingdir/sampledir> -o <profiledir> -r <refdir> -n coverage

The -n parameter enables read-count normalization by gene length. If omitted, no normalization is applied to the gene table.

This profiling step can generate:

  • a species abundance table,
  • an ARG abundance table (full catalogue only),
  • a dbCAN abundance table (full catalogue only),
  • a Gut Metabolic Modules (GMM) abundance table (full catalogue only),
  • a Gut Brain Modules (GBM) abundance table (full catalogue only).

2.16 5. Merging sample outputs

To merge outputs from multiple samples into a single table:

meteor merge -i <profiledir> -r <refdir> -o <mergingdir>

2.17 6. Strain profiling

Meteor2 can profile strains across large metagenomic datasets by identifying sample-specific mutations and projecting them onto MSP reference sequences.

To perform strain profiling:

meteor strain -i <mappingdir/sampledir> -o <straindir> -r <refdir>

To compute mutation rates and build trees between strains:

meteor tree -i <straindir> -o <treedir>

This step generates:

  • a mutation rate matrix,
  • a FASTA file for each reconstructed strain in each sample,
  • a detailed pairwise comparison table for each strain.

2.17.1 Pairwise strain comparison table

Column Description
sample1 First sample in the comparison
sample2 Second sample in the comparison
total_length Total length of the alignment (in bases)
overlap_noN_info_count Number of positions where both samples have minimal information (A, C, G, T or IUPAC codes excluding N, gaps, and ?)
overlap_noIUPAC_info_count Number of positions where both samples have maximal information (strictly A, C, G, T)
overlap_noN_info_pc Percentage of total_length with minimal-information overlap
overlap_noIUPAC_info_pc Percentage of total_length with maximal-information overlap
noN_info_pc_sample1 In sample1, percentage of positions with minimal information
noN_info_pc_sample2 In sample2, percentage of positions with minimal information
noIUPAC_info_pc_sample1 In sample1, percentage of positions with maximal information
noIUPAC_info_pc_sample2 In sample2, percentage of positions with maximal information
distance Genetic distance between samples (0.0 = identical)
distance_category Categorical classification: same_strain, same_subspecies, or divergent

2.17.2 Distance-based classification

Samples are automatically classified according to their genetic distance:

Category Distance threshold Approximate similarity Biological interpretation
same_strain ≤ 0.0001 ≥ 99.99% Same strain / clone
same_subspecies ≤ 0.015 ≥ 97% Same subspecies
divergent > 0.015 < 97% Different lineage

Notes

  • Maximal information: unambiguous nucleotides only (A, C, G, T)
  • Minimal information: includes IUPAC ambiguity codes (R, Y, S, W, K, M, B, D, H, V) but excludes N, gaps (-), and unknown characters (?)
